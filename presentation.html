The Goal

A queryable database — from file chaos to structured queries

Today it looks like this

"I need the plasma proteomics and clinical data for all CAD cases from 2019."

→ Hours of manual file hunting across 28 000 files on AWS S3.

"I found a new batch correction method — what STARNET data does it need as input?"

→ No way to answer without reading the paper and manually checking file by file.

The goal — two queries that should just work

Query 1 — participant lookup
SELECT data_type, file_path FROM starnet WHERE participant_id = 'X'

Returns: proteomics ✓, RNA-seq (liver, MAM) ✓, genotype ✓, clinical ✓

Query 2 — method-driven data fetch
Input: PDF of a new paper.
Output: list of STARNET participants with all required data types present, pulled automatically from the database.

Challenges & Lessons

The main challenge was context, not code

Where the time actually went:

90% — Prompting, context, iteration
10%

10% = output creation (code generated by Claude, reviewed and edited by me)

Challenge — context coordination

I have dozens of files across my laptop with project context: STARNET schemas, BEAT-IT analysis code, AWS infrastructure notes, cohort documentation. Getting Claude to understand the right context for each task was the hardest part — not writing the code itself.

Solution — structured vault

Built a persistent Obsidian knowledge base (architecture docs, session logs, decision records). Every session starts with /load to sync context. Omics Explorer is now cross-linked to STARNET and BEAT-IT architecture docs — it knows where it fits in the bigger pipeline.

Lesson — the 90/10 model

LLM coding is not "write code for me." It is an iterative loop: explain context → get output → review → correct → repeat. The quality of the output is almost entirely determined by the quality of the context. Prompting is the skill.
